# Issue Planning

## Tujuan
Mengubah requirement menjadi GitHub Issues.

## Kapan Digunakan
Digunakan setelah design selesai.

## Input
- `docs/requirements/03-specification.md`
- `docs/design/01-architecture.md`
- `docs/design/02-database-api.md`
- `docs/design/03-ui-flow-wireframe.md`

## Langkah Kerja
1. Baca input.
2. Pecah pekerjaan menjadi issue kecil.
3. Hubungkan issue dengan requirement dan acceptance criteria.
4. Tentukan urutan pengerjaan.

## Output
- `docs/requirements/traceability.md`
- GitHub Issues

## Aturan
- Minimal 10 issue.
- Setiap issue harus bisa dikerjakan dan diuji.

## Quality Check
- Issue tidak terlalu besar.
- Setiap issue punya test atau acceptance check.
- Traceability diperbarui.

## Kondisi Gagal
Berhenti jika design belum tersedia.

## Human Review
Mahasiswa menyetujui urutan issue sebelum coding.
