## 2. Review Status and Boundary

This document defines the UI design for the Campus Service Request and Maintenance System. It translates approved requirements, architecture, and database/API design into screens, navigation, role-based views, form behavior, list/table behavior, detail behavior, UI states, action permissions, textual wireframes, and requirement-to-UI mapping.

| Architecture Design | Approved and merged to `development` | Source baseline is `docs/design/01-architecture.md`. |

### Boundary

This UI design includes:

- screen list and role-based entry points;
- navigation flow;
- form fields and validation behavior;
- table/list, search, and filter behavior;
- report detail behavior;
- comments/notes and status history presentation;
- UI action permissions;
- loading, empty, error, forbidden, validation, success, and not found states;
- textual wireframes;
- requirement-to-UI mapping.

This UI design does not create GitHub issues, automated tests, React/TypeScript implementation, backend/API implementation, migration files, migration execution, deployment config, or Cloudflare deployment.

Deferred optional features remain out of required scope:

- Upload photo.
- Email notification.
- Google login.
- Room QR code.
- AI categorization.
- Spare-part inventory.
- Vendor management.

## 3. UI Design Goals

| Goal ID | Goal | Requirement Basis |
|---|---|---|
| UI-GOAL-001 | Make the full lifecycle visible and understandable from Submitted to Closed. | `FR-001` to `FR-014`, `BR-001`, `CR-001` |
| UI-GOAL-002 | Give each actor a clear role-specific path without adding a production authentication design. | `NFR-002`, `BR-003`, `BR-004`, `CR-003` |
| UI-GOAL-003 | Keep forms, tables, details, and actions simple enough for a student project. | `NFR-001`, `NFR-005` |
| UI-GOAL-004 | Surface status history and append-only comments/notes as report context. | `FR-012`, `FR-013`, `CR-002` |
| UI-GOAL-005 | Support simple management visibility without advanced analytics. | `FR-015`, `NFR-007` |

### Visual Direction

The visual style should be a simple educational admin/workflow interface: clear labels, readable tables, compact detail panels, status badges, restrained colors, and predictable buttons. Final color tokens, typography, component library, CSS, or pixel-perfect layout are implementation-stage decisions and are not finalized here.

## 4. Actors and Role-Based Entry Points

| Role | Entry Point | Primary UI Needs | Out of Scope |
|---|---|---|---|
| Reporter | Reporter workspace | Create report, view own/relevant reports, open detail, view status/history, add comments. | Photo upload, Google login, email notification. |
| Administrator | Admin workspace | Review submitted reports, set category/priority, assign technician, close/reopen, add notes. | Multi-level admin approval, duplicate merge feature. |
| Technician | Technician workspace | View assigned tasks, accept assigned work, update status, mark resolved, add progress notes. | Reject/reassign workflow. |
| Facility Manager | Manager dashboard | View summary counts and recent reports. | Advanced analytics, exports, inventory, vendor reporting. |

Because `CR-003` keeps the role boundary simple and educational, the implementation now uses a demo login page that asks for username, role, and password, then matches those values against the seeded `app_users` demo actor records. For the current demo data, the seeded actor ID is used as both username and password. The page stores the resolved actor context in frontend storage. This replaces the earlier in-header role selector for the demo flow only; it is not production authentication and does not add Google login, registration, password reset, or email verification.

## 5. Navigation Flow

### UI-NAV-01: Role-Aware App Shell

```text
Role entry / current role indicator
  -> Reporter: New Report, My Reports
  -> Administrator: Review Queue, All Reports
  -> Technician: My Tasks
  -> Facility Manager: Dashboard
```

### UI-NAV-02: Reporter Flow

```text
Reporter Workspace
  -> New Report Form
    -> success: Report Detail
    -> validation error: stay on form and show field errors
  -> My Reports
    -> Report Detail
      -> Add Comment
      -> View Status History
```

### UI-NAV-03: Administrator Flow

```text
Admin Workspace
  -> Review Queue / All Reports
    -> Report Detail
      -> Review Submitted Report
      -> Set Category and Priority
      -> Assign Technician
      -> Add Note
      -> Close Resolved Report
      -> Reopen Resolved/Closed Report to Under Review
```

### UI-NAV-04: Technician Flow

```text
Technician Workspace
  -> My Tasks
    -> Task Detail
      -> Accept Assigned Work
      -> Mark In Progress
      -> Add Progress Note
      -> Mark Resolved
```

### UI-NAV-05: Facility Manager Flow

```text
Manager Dashboard
  -> Summary Counts
  -> Recent Reports
    -> Report Detail (read-only management view)
```

## 7. Forms and Validation

### UI-FORM-01: New Report Form

| Field | Control | Required | Validation / Behavior |
|---|---|---|---|
| Title | Text input | Yes | Required; show inline error if empty. |
| Description | Textarea | Yes | Required; explain that the reporter should describe the problem clearly. |
| Location | Text input | Yes | Required; accepts campus building, room, or area. |
| Category | Select | Yes | Options: Internet, AC, Peralatan Kelas, Kebersihan, Laboratorium, Lainnya. |
| Reporter name | Text input | Yes | Required by `FR-002`. |
| Reporter contact | Text input | Yes | Required by `FR-002`; label should allow email, phone, or campus contact. |

Submit behavior:

- Primary action: `Submit Report`.
- On success: show created request number and navigate to `UI-04` detail.
- On validation error: keep entered values and show field-level errors.
- On backend error: show non-field error and allow retry.

### UI-FORM-02: Admin Review and Classification Form

| Field / Action | Control | Required | Validation / Behavior |
|---|---|---|---|
| Review action | Button | Only for `SUBMITTED` | Moves report to `UNDER_REVIEW`. |
| Category | Select | Yes when saving classification | Same approved category set as `UI-FORM-01`. |
| Priority | Select | Yes when saving classification | Low, Medium, High. |
| Review note | Textarea | Optional | Saved through comments/notes as append-only context. |

### UI-FORM-03: Assignment Form

| Field | Control | Required | Validation / Behavior |
|---|---|---|---|
| Technician | Select | Yes | Populated from active technicians. |
| Assignment reason/context | Textarea | Optional | Helps explain assignment in history/note context. |

Submit behavior:

- Primary action: `Assign Technician`.
- On success: status becomes `ASSIGNED` and technician appears on detail.
- If no technician is selected: show validation error.

### UI-FORM-04: Comment/Note Form

| Field | Control | Required | Validation / Behavior |
|---|---|---|---|
| Comment/note body | Textarea | Yes | Required; append-only after submission. |
| Type label | Fixed by context or simple select | Yes | Reporter uses comment; staff may use comment or note if implementation keeps both `COMMENT` and `NOTE`. |

Behavior:

- Comments/notes cannot be edited or deleted in the initial scope.
- Saved entries appear in chronological order with author role and timestamp.
- All entries are visible to authorized users who can view the report, per `CR-002`.

### UI-FORM-05: Technician Progress Form

| Field / Action | Control | Required | Validation / Behavior |
|---|---|---|---|
| Accept work | Button | For assigned technician on `ASSIGNED` | Records acceptance before progress starts. |
| Work status | Button or select with constrained actions | Yes | Allowed UI actions: mark `IN_PROGRESS` from assigned/accepted work, then mark `RESOLVED` from `IN_PROGRESS`. |
| Progress note | Textarea | Optional for progress, recommended for resolved | Stored as append-only context. |

### UI-FORM-06: Close/Reopen Form

| Field / Action | Control | Required | Validation / Behavior |
|---|---|---|---|
| Close report | Button with confirmation | For Administrator on `RESOLVED` | Moves status to `CLOSED`. |
| Reopen reason | Textarea | Yes for reopen | Required by `CR-001` and `VAL-DBAPI-009`. |
| Reopen report | Button with confirmation | For Administrator on `RESOLVED` or `CLOSED` | Moves status to `UNDER_REVIEW`. |

## 8. Tables, Lists, Search, and Filters

### UI-LIST-01: Shared Report List

Default columns:

| Column | Notes |
|---|---|
| Request number | Clickable link to `UI-04`. |
| Title | Short issue summary. |
| Location | Campus location. |
| Category | Human-readable category label. |
| Priority | Low, Medium, High badge. |
| Status | Lifecycle status badge. |
| Assigned technician | Empty label when unassigned. |
| Created date | Used for recent activity and default ordering. |

Search/filter controls:

| Control | Behavior |
|---|---|
| Keyword search | Searches practical text fields such as request number, title, description, location, or reporter info depending on API support. |
| Status filter | Submitted, Under Review, Assigned, In Progress, Resolved, Closed. |
| Category filter | Internet, AC, Peralatan Kelas, Kebersihan, Laboratorium, Lainnya. |
| Priority filter | Low, Medium, High. |
| Location filter | Text filter for building, room, or area. |
| Assigned technician filter | Admin/manager view only if useful; uses technician list. |

Role-specific list behavior:

| Role | List Scope |
|---|---|
| Reporter | Shows reports relevant to the reporter. Exact identity mechanism remains open, so implementation may start with role/demo scope. |
| Administrator | Shows all reports, with default quick view for Submitted and Under Review. |
| Technician | Uses `UI-07` task list for assigned reports. |
| Facility Manager | Shows recent reports from dashboard and may open read-only detail. |

Empty state:

- No reports yet: explain that no service requests exist.
- No filter results: explain that no reports match current search/filter and offer clear filters action.

## 9. Detail Views, Status History, and Comments/Notes

### UI-DETAIL-01: Report Detail Layout

Report detail should show:

- request number;
- title;
- current status badge;
- priority badge;
- category;
- location;
- description;
- reporter name and contact;
- assigned technician if available;
- important timestamps;
- role-specific action panel;
- comments/notes;
- status history timeline.

### UI-COMP-01: Lifecycle Indicator

Display the approved lifecycle in order:

```text
Submitted -> Under Review -> Assigned -> In Progress -> Resolved -> Closed
```

Rules:

- Highlight the current status.
- Completed previous statuses can appear as completed steps if history exists.
- Reopened reports show current status `Under Review`, with status history explaining the reopen.
- Do not add new statuses without an approved change request.

### UI-COMP-02: Status History Timeline

Each history entry shows:

- from status;
- to status;
- changed by role;
- reason/context when available;
- timestamp.

History is read-only in the UI.

### UI-COMP-03: Comments/Notes Timeline

Each comment/note entry shows:

- author role;
- comment type;
- body;
- timestamp.

Rules:

- Append-only.
- No edit/delete action in initial scope.
- Visible to authorized users who can view the related report.

## 10. UI Action Permissions

| UI Action ID | Action | Reporter | Administrator | Technician | Facility Manager | Status Conditions |
|---|---|---:|---:|---:|---:|---|
| UI-ACT-01 | Create report | Yes | No | No | No | Creates `SUBMITTED`. |
| UI-ACT-02 | View report list | Yes | Yes | Limited | Yes | Scope depends on role. |
| UI-ACT-03 | View report detail | Yes | Yes | Limited | Yes | Must be authorized to view report. |
| UI-ACT-04 | Review submitted report | No | Yes | No | No | Only from `SUBMITTED` to `UNDER_REVIEW`. |
| UI-ACT-05 | Set category/priority | No | Yes | No | No | Administrator action in review/admin detail. |
| UI-ACT-06 | Assign technician | No | Yes | No | No | Moves eligible report to `ASSIGNED`. |
| UI-ACT-07 | Accept assigned work | No | No | Yes | No | Assigned technician only. |
| UI-ACT-08 | Mark in progress | No | No | Yes | No | Assigned technician after acceptance. |
| UI-ACT-09 | Mark resolved | No | No | Yes | No | Assigned technician from `IN_PROGRESS`. |
| UI-ACT-10 | Add comment/note | Yes | Yes | Yes | No | Authorized report viewers; manager read-only by default. |
| UI-ACT-11 | Close report | No | Yes | No | No | Only from `RESOLVED`. |
| UI-ACT-12 | Reopen report | No | Yes | No | No | From `RESOLVED` or `CLOSED` to `UNDER_REVIEW`; reason required. |
| UI-ACT-13 | View dashboard | No | No | No | Yes | Facility Manager only. |

Disabled or hidden actions:

- Prefer hiding actions that are irrelevant to a role.
- Prefer disabled buttons with short explanation when the role is correct but the current status does not allow the action.
- If the API returns `FORBIDDEN_ACTION` or `INVALID_STATUS_TRANSITION`, show a clear error and refresh report detail.

## 11. Loading, Empty, Error, and Success States

| UI State ID | State | Applies To | Behavior |
|---|---|---|---|
| UI-STATE-01 | Loading | Lists, detail, dashboard, technician task list | Show loading text or skeleton area without changing filters/form values. |
| UI-STATE-02 | Empty list | Report list, task list | Show a plain empty message and keep search/filter controls available. |
| UI-STATE-03 | Empty dashboard | Dashboard | Show zero counts and an empty recent reports message without error. |
| UI-STATE-04 | Validation error | Forms | Show field-level messages from `VALIDATION_ERROR` details. |
| UI-STATE-05 | Forbidden action | Role-sensitive actions | Explain that the current role cannot perform the action. |
| UI-STATE-06 | Invalid transition | Lifecycle actions | Explain that the report status changed or the action is not valid from the current status. Refresh detail. |
| UI-STATE-07 | Not found | Detail view | Explain that the report was not found or is unavailable. Offer navigation back to list. |
| UI-STATE-08 | Success feedback | Create, review, assign, status update, comment, close, reopen | Show concise confirmation and update the current view. |
| UI-STATE-09 | Internal/server error | Any API-backed view | Show retry path without losing user-entered form text where practical. |
